# Domain Docs

工程技能在分析代码库时应如何读取本项目的领域文档。

## 探索前先读这些

- **`CONTEXT.md`**（项目根目录），或
- **`CONTEXT-MAP.md`**（项目根目录，如果存在的话）——它指向每个上下文的 `CONTEXT.md`。读取所有与当前主题相关的上下文文件。
- **`docs/adr/`**——读取与你即将工作的领域相关的 ADR。多上下文仓库还需检查 `src/<context>/docs/adr/` 下的上下文级决策。

如果以上文件不存在，**静默跳过**。不要报告缺失，也不要主动建议创建。生产者技能（`/grill-with-docs`）会在术语或决策实际被确定时按需创建它们。

## 文件结构

单上下文仓库（本项目的布局）：

```
/
├── CONTEXT.md
├── docs/adr/
│   ├── 0001-xxx.md
│   └── 0002-xxx.md
└── src/
```

## 使用术语表中的词汇

当你的输出中涉及领域概念（issue 标题、重构提案、假设、测试名称）时，使用 `CONTEXT.md` 中定义的术语。不要偏离到术语表明确避免的同义词。

如果你需要的概念尚未出现在术语表中，这是一个信号——要么你在创造项目未使用的语言（重新考虑），要么确实存在缺口（记下来供 `/grill-with-docs` 使用）。

## 标记 ADR 冲突

如果你的输出与已有 ADR 矛盾，明确提出来，而不是静默覆盖：

> _与 ADR-0007（xxx）矛盾——但值得重新讨论，因为……_
